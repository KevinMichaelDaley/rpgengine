# Undo support for bone transforms and operations

Bone transforms (move, rotate in both prefab mode and regular edit mode) currently bypass the undo system entirely. The gizmo applies bone pose changes directly via `per_bone_gizmo_apply_drag/rotate` and the TUI `move`/`rotate` commands check for bone selection and apply locally without recording undo entries.

## Current state

- `scene_gizmo_bone.c` / `scene_gizmo_bone_draw.c` — bone gizmo rendering and interaction
- `scene_gizmo_bone_apply.c` — applies bone transforms directly to skeleton joint data
- `scene_input.c` lines ~650-695 — TUI move/rotate intercepts for bone selection, applies locally, no undo
- `bone_pose_store.c` — per-entity bone pose overrides (regular edit mode)
- No `EDIT_CMD_TYPE_BONE_*` variants exist

## Deliverables

- Add `EDIT_CMD_TYPE_BONE_MOVE` and `EDIT_CMD_TYPE_BONE_ROTATE` to `edit_cmd_type_t`
- Record undo entries when bone transforms are applied (both prefab and regular mode):
  - Store entity_id, bone_index, and inverse delta (position or quaternion)
  - For prefab mode: inverse modifies `skeleton_def_t.joints[bone].rest_local`
  - For regular mode: inverse modifies `bone_pose_block_t` override
- Extend `undo_apply.c` to handle bone command types
- Extend `undo_conflict_key_t` with bone_index for bone-level conflict detection
- Bone undo entries should use the entity_id + bone_index as conflict key (two bone edits on the same bone conflict, different bones don't)
- Ctrl+Z should reverse bone transforms just like entity transforms
- Tests for bone move undo, bone rotate undo, mixed bone+entity undo ordering

## Dependencies

- rpg-obka — Bone editing in regular edit mode (in progress; defines the bone pose override storage that undo must interact with)
